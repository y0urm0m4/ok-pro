# Правила работы AI-агента

## Базовые принципы

1. **Сначала читай контекст, потом пиши код.** Перед задачей просмотри `PROJECT.md`, `ARCHITECTURE.md`, `DESIGN_SYSTEM.md` и релевантные файлы в проекте.
2. **Mobile-first.** Всегда сначала мобильная вёрстка, потом desktop. Основной трафик — Instagram-таргет с телефонов.
3. **Маленькие шаги.** Один блок = один коммит. Не пиши весь лендинг в одном сообщении.
4. **TypeScript strict.** Никаких `any`, никаких `// @ts-ignore` без комментария-обоснования.
5. **Не изобретай дизайн-решения наугад.** Если нужен цвет, отступ, размер — бери из `DESIGN_SYSTEM.md` и `tailwind.config.ts`.

## Что МОЖНО делать без подтверждения

- Создавать компоненты по структуре из `ARCHITECTURE.md`
- Использовать токены из дизайн-системы
- Добавлять утилиты в `lib/`
- Устанавливать пакеты из явно одобренного списка в `STACK.md`
- Рефакторить, если соблюдается контракт компонента

## Что НЕЛЬЗЯ без подтверждения

- Менять стек (React, Next.js, Tailwind — фиксированы)
- Подключать сторонние UI-киты (никаких MUI, Ant Design, Chakra и т.д.)
- Подключать аналитику, пиксели, чужие скрипты
- Менять структуру папок из `ARCHITECTURE.md`
- Добавлять API-ключи, секреты в код (только через `.env`)
- Подключать CMS на этапе MVP (блог — это этап 2)
- Реализовывать функционал «потому что красиво» если его нет в `ROADMAP.md`

## Как писать код

### Компоненты
- Server Components по умолчанию. `'use client'` — только когда реально нужен (state, эффекты, обработчики)
- Один компонент = один файл, имя файла `PascalCase.tsx`
- Пропсы типизировать через `interface ComponentNameProps`
- Дефолтный экспорт для страниц/layout, named export для компонентов

### Стили
- Только Tailwind. Никакого CSS-in-JS, никаких `.module.css` без крайней необходимости
- Длинные классы — через `clsx` или `cn` (utility из `lib/cn.ts`)
- Кастомные значения — только через токены в `tailwind.config.ts`, не пиши `text-[#a83c3c]` инлайном

### Картинки
- Только через `next/image`
- Всегда указывать `alt`, `width`, `height` или `fill` + `sizes`
- Hero-изображение: `priority`, остальные — lazy по умолчанию

### Доступность
- Семантические теги: `<header>`, `<nav>`, `<main>`, `<section>`, `<footer>`
- Один `<h1>` на страницу
- Кнопки — `<button>`, ссылки — `<a>` или `<Link>`. Не имитируй кнопки `<div onClick>`
- Все интерактивные элементы фокусируемы клавиатурой
- `aria-label` на иконочных кнопках без видимого текста

### Производительность
- Тяжёлые компоненты (slider, аккордеон, modal) — `dynamic` импорт с `ssr: false` где уместно
- Шрифты через `next/font`
- Никаких `<script>` от третьих сторон без согласования

## Коммиты

Формат: `тип: краткое описание`

Типы: `feat`, `fix`, `style`, `refactor`, `docs`, `chore`

Примеры:
- `feat: hero-секция с CTA`
- `feat: форма заявки с валидацией`
- `style: анимация появления карточек услуг`
- `fix: переполнение текста в FAQ на mobile`

## Когда задавать вопросы

Задавай вопрос, если:
- В контексте противоречие
- Нужен текст/контент, которого нет в `CONTENT.md`
- Требуется решение, которое влияет на структуру или дизайн-систему
- Срабатывает чувство «здесь нужно уточнить» — лучше спросить

## Когда НЕ задавать вопросы

Не спрашивай по мелочам, которые покрыты документацией: «какой шрифт?», «какой основной цвет?», «куда положить компонент?» — это всё в файлах документации.
